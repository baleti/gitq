#+title: gitq — a typed, categorical query language for git
#+author: baleti

One query language over git's object graph: describe the data you want by
composing small, typed operations, and end the pipeline with what should
happen to the result.

#+begin_example
  gitq 'commits where author alice take 5 /show'
  gitq 'HEAD via parent* where message "fix" /count'
  gitq 'commits pickaxe "some-function" first /worktree'
  gitq 'HEAD via tree.entries.history sort -date first /copy'
#+end_example

A pipeline is =source [step...] [/terminal]=: a source produces typed frames
(=commits=, =branches=, =tags=, =refs=, =worktrees=, =blobs=, =HEAD=, any
ref), steps transform them (=via= morphism paths like
=parent[0].tree.entries[Blob]= compose and type-check segment by segment;
=where= / =sort= / =pick= / =take= / =grep= / =pickaxe= behave as their
SQL/PRQL/jq counterparts), and a =/terminal= acts on the result (=/show=
=/copy= =/count= =/branch-off= =/reword= =/remove= =/worktree= ...).

Everything is validated at parse time against the frame shape actually
flowing through the pipeline — unknown fields, operators that don't fit a
field's type, or morphisms applied to the wrong object are immediate,
specific errors, never silent empty results.  There is no explicit
=contains= keyword: a value right after a field with no recognized
operator between them gets the field type's implicit operator — substring
match for text-shaped fields (=where author bal=, =where date 2026-07=),
equality for numbers (=where parents-count 2=).

The full design — the typed categorical model, grammar, and extension
guide — is in [[file:doc/gitq.org][doc/gitq.org]]; [[file:doc/examples.org][doc/examples.org]] is a set of worked examples
with real captured outputs, including the pickaxe + =diff.lines= workflow
for seeing matched diff content inline.

* History

gitq began as Emacs Lisp inside the [[https://github.com/baleti/git-branch-off][git-branch-off]] package
(=git-branch-off-gitq.el=), where the language, its registries, and its
fail-loud type discipline were designed and battle-tested.  This repository
is the standalone rewrite in Haskell: the language is the same, the
completion engine and parser live in one binary, and Emacs and zsh become
thin integrations over it.

* Installation

** Prebuilt binary (no toolchain needed)

Every tagged release ships a Linux x86_64 binary (native gix backend
included), built and tested by [[file:.github/workflows/release.yml][the release workflow]]:

#+begin_src sh
  curl -L -o ~/.local/bin/gitq \
    https://github.com/baleti/gitq/releases/latest/download/gitq-x86_64-linux
  chmod +x ~/.local/bin/gitq
#+end_src

The Emacs integration needs no manual step at all: on first use, if no
=gitq= is found on =exec-path=, it downloads the release binary into
the package's own directory automatically — so installing the straight
recipe is the whole setup, and removing the recipe removes the binary
with it.  A user-installed binary on PATH always takes precedence, and
=M-x gitq-install-binary= forces a re-download.

** From source

Requires GHC ≥ 9.4 and cabal; the native backend additionally needs a
Rust toolchain.

#+begin_src sh
  git clone https://github.com/baleti/gitq ~/gitq
  cd ~/gitq
  make install           # subprocess-git only, no Rust needed
  make install-native    # with the in-process gix backend
#+end_src

Both install to =~/.local/bin= (override with =BINDIR=...=).  Run the
test suite with =cabal test= (or =make test-native=).

** Optional: the Rust in-process git backend

With a Rust toolchain installed, =make install-native= (or =cabal
install -fnative=) links a small Rust static library (=native/=, gix)
into the same =gitq= binary.  Commit-batch
resolution and =parent*=/=parent+= closures then run in-process instead
of through git subprocesses — same output (the Rust side emits records in
the exact byte format of gitq's =git log= format string, parsed by the
same function), with the subprocess path kept as an automatic fallback on
any native-side failure.  =GITQ_NO_NATIVE=1= forces the fallback at
runtime, so one binary can A/B itself.

Where it matters: fixed-cost latency on small queries (a small repo's
=parent*= closure drops ~52 ms → ~27 ms — the live-preview path), and
deep-history walks: gix enumerates git/git's 81,470-commit closure in
0.08 s when a commit-graph is present (=git commit-graph write
--reachable=; git maintains it automatically on gc) — 20× faster than
libgit2 and faster than =git rev-list= itself (measured in
=native/examples/walk.rs= vs =walk_gix.rs=).  End-to-end bulk queries are
currently bounded by frame construction on the Haskell side rather than
the backend; the boundary is in place for when that moves.

* CLI usage

#+begin_example
  gitq '<pipeline>'             run a pipeline (from anywhere inside a repo)
  gitq --preview '<pipeline>'   run source and steps, never apply a terminal
  gitq --sexp '<pipeline>'      print frames as Emacs Lisp plists
  gitq --complete '<prefix>'    completion candidates for a partial pipeline
#+end_example

* Emacs integration

[[file:integrations/emacs/gitq.el][integrations/emacs/gitq.el]] provides =M-x gitq= with per-token contextual
completion (candidates and annotations come from the binary, so they can
never drift from the parser), a debounced live preview of results while
typing, and a results buffer: =RET= visits the object at point (via Magit
when available), =b= branches off, =c= copies the SHA.  Re-invoking =gitq=
from a results buffer pre-fills the minibuffer with the query that built it.

#+begin_src emacs-lisp
;; Doom Emacs, packages.el
(package! gitq
  :recipe (:host github :repo "baleti/gitq"
           :files ("integrations/emacs/gitq.el")
           :branch "main"
           :depth full))
#+end_src

The =gitq= binary must be on =exec-path= (or set =gitq-executable=).

* Zsh completion

Entirely per-user — completions live in any directory on your =$fpath=,
no root or system paths involved:

#+begin_src sh
  cd ~/gitq && make install-zsh
#+end_src

If your shell already has a writable =$HOME= directory on =$fpath=
(oh-my-zsh, zinit, antidote, and friends all add one), the link goes
there and *no zshrc changes are needed* — just refresh the completion
cache (=rm -f ~/.zcompdump && exec zsh=).  Stock zsh has no per-user
fpath directory, so in that case the link goes to the XDG data dir
(=~/.local/share/zsh/completions= — completion functions are data,
mirroring the system =site-functions= convention, no new dotfile
directory in =$HOME=) and the installer prints the two required
=~/.zshrc= lines:

#+begin_src sh
  fpath=(~/.local/share/zsh/completions $fpath)
  autoload -Uz compinit && compinit
#+end_src

Override the destination with =make install-zsh ZSH_COMP_DIR=...= —
e.g. =$ZDOTDIR/completions= if you keep zsh config under
=~/.config/zsh=.

Completion inside the pipeline argument delegates to =gitq --complete=, so
zsh, Emacs, and the parser always agree on what is valid where.

* Bash completion

Same delegation model as zsh — no grammar is duplicated, both call
=gitq --complete=:

#+begin_src sh
  cd ~/gitq && make install-bash
#+end_src

That symlinks =integrations/bash/gitq.bash= into
=~/.local/share/bash-completion/completions/gitq=.  Most distributions'
=bash-completion= package auto-sources that directory; if yours does not,
the installer prints the one line to add to =~/.bashrc=:

#+begin_src sh
  source ~/.local/share/bash-completion/completions/gitq
#+end_src

Override the destination with =make install-bash BASH_COMP_DIR=...=.
Completion works both with the pipeline quoted (=gitq 'commits wh<TAB>= →
=gitq 'commits where'=) and unquoted (=gitq commits wh<TAB>=); bash lists
plain candidates rather than the categorized menu zsh's =compsys= draws,
which is a display nicety, not required behavior.

* Differences from the Emacs Lisp original

The language is unchanged; a few behaviors are adapted to a real terminal,
each failing loud rather than degrading silently:

- =/copy= uses =wl-copy= / =xclip= / =xsel= / =pbcopy= (errors if none is
  installed) instead of the Emacs kill ring; =/insert= prints the SHA.
- =/branch-off= and =/mark= require their name/label argument (the original
  prompted in the minibuffer); =/amend= and =/commit= without a message
  open =$EDITOR=, as git normally would.
- =/remove= is implemented directly (=git rebase --onto=) with clean-tree
  and is-ancestor guards; =/reword= currently only rewords HEAD.
- =/regex/= patterns and the =regex= operator are POSIX ERE (the original
  used Emacs regexp syntax); =git grep= and pickaxe patterns were always
  ERE in both.
- =where date after/before= parses dates properly (timezone-aware) instead
  of comparing lexically.
